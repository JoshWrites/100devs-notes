## 1. Prerequisites

### GitHub CLI

Check if `gh` is installed:
```bash
gh auth status
```

If not installed:
```bash
sudo apt install gh
gh auth login
```

Choose **HTTPS** for the auth protocol (simplest, no SSH key management needed).

### Node.js & npm

```bash
sudo apt install nodejs npm
```

Verify:
```bash
node --version
npm --version
```

### LibreOffice (required for RTF import)

Already installed on the primary dev machine. Verify:
```bash
libreoffice --version
```

## 2. Initial Setup

### Clone the repo

```bash
git clone https://github.com/JoshWrites/100devs-notes.git
cd 100devs-notes
```

### Install dependencies

```bash
npm install
```

This installs:
- `js-yaml` — YAML frontmatter parsing in build script
- `marked` — Markdown → HTML in build script
- `sanitize-html` — HTML sanitization at build time
- `mammoth` — DOCX → HTML for bulk import
- `turndown` — HTML → Markdown for bulk import
- `puppeteer` — optional (Coda migration scripts only), skipped in CI

## 3. Repo Structure

```
notes/                      Markdown notes, organized by author
  {author-slug}/
    YYYY-MM-DD.md

import/                     Drop source files here for bulk import
  manifest.json             Maps filenames to author/date/tags metadata
  .gitignore                Source files are gitignored — only manifest is tracked

scripts/
  build.js                  Builds notes/ → dist/ and generates index.json
  import-notes.js           Bulk import tool (RTF, DOCX, HTML, MD)
  validate-notes.js         PR validation — checks frontmatter on changed notes
  retag-notes.js            Re-analyze and update tags across all notes

.github/
  CODEOWNERS                Auto-assigns @Nili-L as reviewer on all PRs
  pull_request_template.md  Pre-filled checklist shown on every new PR
  workflows/
    deploy.yml              Builds index.json and optionally deploys on push to main
    validate-notes.yml      Validates note frontmatter on PRs touching notes/

dist/                       Built output (gitignored, generated by npm run build)
  index.json
  robots.txt
  sitemap.xml               Only generated when SITE_URL env var is set

index.json                  Root-level copy of dist/index.json committed to repo
                            (served directly by the viewer via fetch)

viewer.html                 The notes viewer single-page app
TEMPLATE.md                 Starter template for new notes
```

## 4. Security Fixes Applied

These were identified in a code review and fixed in commit `e4b61e7`.

### Command injection in build script

**Before** (`build.js`):
```js
const out = execSync(`git log -1 --format=%an -- "${rel.replace(/"/g, '\\"')}"`)
```

**After** — uses `spawnSync` with an args array (no shell interpolation):
```js
const result = spawnSync('git', ['log', '-1', '--format=%an', '--', rel], {
  cwd: REPO_ROOT, encoding: 'utf8'
});
```

### Hand-rolled YAML parser replaced

**Before**: custom parser that failed on quoted strings and complex values.

**After**: uses `js-yaml` (already a dependency):
```js
const yaml = require('js-yaml');
const frontmatter = yaml.load(match[1]) || {};
```

### XSS prevention

Two layers of sanitization:

1. **Build time** (`build.js`) — HTML from `marked` is sanitized with `sanitize-html` before being stored in `index.json`:
```js
const sanitizeHtml = require('sanitize-html');
const html = sanitizeHtml(marked.parse(body), {
  allowedTags: sanitizeHtml.defaults.allowedTags.concat(['img']),
  allowedAttributes: { ... },
});
```

2. **Runtime** (`viewer.html`) — DOMPurify sanitizes before setting `innerHTML`:
```html
<script src="https://cdn.jsdelivr.net/npm/dompurify@3/dist/purify.min.js"></script>
```
```js
document.getElementById('noteBody').innerHTML = DOMPurify.sanitize(note.html);
```

### GitHub Actions permissions

Added explicit `permissions: contents: write` to `deploy.yml` rather than relying on defaults.

### Puppeteer removed from CI installs

Moved `puppeteer` from `devDependencies` to `optionalDependencies`. CI uses:
```bash
npm ci --omit=optional
```

## 5. Viewer Features Added

All changes live in `viewer.html`.

### Full-text search

Search input in the sidebar filters across note content, author names, dates, and tags in real time. Search and tag filters compose together.

### All Notes view

"All notes" entry at the top of the writer list shows every note as a flat list sorted newest-first, with date, author, and tags visible at a glance.

### Deep linking / URL routing

Opening a note updates the URL hash to `#author-slug/YYYY-MM-DD`. Sharing or bookmarking that URL reopens the exact note. Works across page loads and browser back/forward.

### Open Graph + Twitter meta tags

When a note is opened, the page title, OG title, description, and URL are updated dynamically. Sharing a note URL on social media shows the note author and date in the preview card.

### Mobile layout

On screens under 640px, the sidebar is hidden by default. A hamburger button in the page header opens it as a slide-in panel with an overlay backdrop.

### European date format

All dates display as `DD/MM/YYYY` throughout the viewer. The build stores dates in ISO format; the viewer converts them at render time:
```js
function formatDate(iso) {
  var parts = String(iso).slice(0, 10).split('-');
  if (parts.length !== 3) return iso;
  return parts[2] + '/' + parts[1] + '/' + parts[0];
}
```

### Print styles

`@media print` hides the sidebar and chrome, shows the note body cleanly, and prints link URLs inline:
```css
@media print {
  .note-body a::after { content: " (" attr(href) ")"; }
}
```

### robots.txt and sitemap.xml

Generated automatically during `npm run build`. Sitemap is only generated when `SITE_URL` is set:
```bash
SITE_URL=https://levinelabs.co.il/100devs/notes npm run build
```

### Source URL attribution

Notes can include an optional `source` field in their frontmatter linking to the original recording, document, or resource the note came from. When present, it appears as a clickable **source** link in the note meta line next to the date and tags.

In a note file:
```yaml
---
tags:
  - the-hunt
  - networking
author: Anny Levine
date: 2024-05-21
source: https://example.com/recording-link
---
```

Rules:
- Must be a valid `http://` or `https://` URL — anything else is rejected by the CI validator
- Omitting the field entirely is fine; it's optional
- The build script validates the URL and silently drops malformed values rather than failing the build

## 6. In-Browser Editing

On branch `in-browser-editing`. Adds GitHub-authenticated editing and note creation directly in the viewer.

### How authentication works

- "Sign in to edit" button appears in the top-right of the header
- Prompts for a GitHub Personal Access Token with `repo` scope
- Token is validated against the GitHub `/user` API
- Stored in `sessionStorage` (cleared when the browser closes)
- Restores session on page reload

**To create a PAT:** GitHub → Settings → Developer settings → Personal access tokens → Generate new token → check `repo`

### Editing an existing note

1. Sign in with your PAT
2. Open any note — an **edit** button appears in the note header
3. Click it — the editor panel slides in from the right
4. Write in Markdown (default) or switch to HTML input — HTML is converted to Markdown via Turndown before saving
5. Use the **Preview** tab to see rendered output
6. Click **Save** — commits directly to `main` via the GitHub API
7. CI rebuilds `index.json` within seconds

### Creating a new note

1. Sign in with your PAT
2. Click **+ new note** at the bottom of the sidebar
3. Fill in: author slug (folder name), display name, huddle date, tags, and optionally a source URL
4. Write content in the editor
5. Click **Save** — auto-generates YAML frontmatter (including `source` if provided) and commits to `main`

### Format toggle

| Input format | What happens on save |
|---|---|
| Markdown | Saved as-is |
| HTML | Converted to Markdown via Turndown, then saved |

## 7. Bulk Import Tool

For importing many notes at once from mixed file formats. Especially useful for the initial re-import after fixing wrong attributions.

### Supported formats

| Format | Conversion method |
|---|---|
| `.md` | Pass-through (frontmatter stripped and regenerated) |
| `.html` | Turndown → Markdown |
| `.rtf` | LibreOffice → HTML → Turndown → Markdown |
| `.docx` | mammoth → HTML → Turndown → Markdown |

### Step 1 — Fill in the manifest

Edit `import/manifest.json` to map each source filename to its correct metadata:

```json
{
  "may21-huddle.rtf": {
    "authorSlug": "annylevine",
    "author": "Anny Levine",
    "date": "2024-05-21",
    "tags": ["the-hunt", "networking"],
    "source": "https://example.com/recording-link"
  },
  "june05-notes.docx": {
    "authorSlug": "michaelgovaerts",
    "author": "Michael Govaerts",
    "date": "2024-06-05",
    "tags": ["the-hunt", "resume"]
  }
}
```

`authorSlug` must be lowercase letters, numbers, and hyphens only. `date` must be `YYYY-MM-DD`. Tags must be from the approved list (see section 11). `source` is optional — include it when you have a URL to the original recording or document.

### Step 2 — Drop files into the import/ directory

```bash
cp ~/Downloads/my-notes.rtf import/
cp ~/Downloads/other-notes.docx import/
```

Source files are gitignored — only `manifest.json` is tracked.

### Step 3 — Dry run

Preview what will happen without writing any files:
```bash
npm run import:dry
```

### Step 4 — Run the import

```bash
npm run import
```

This converts all files, writes them into `notes/`, and runs `npm run build` to regenerate `index.json`.

### Step 5 — Commit and push

```bash
git add notes/ index.json
git commit -m "Import notes with corrected attributions"
git push
```

CI will pick up the push, rebuild `index.json`, and the notes appear on the site within minutes.

### Running with a custom directory

```bash
node scripts/import-notes.js --dir /path/to/my/files
```

The manifest is always read from `import/manifest.json` regardless of `--dir`.

## 9. CI/CD

### deploy.yml — triggers on push to main

Runs when `notes/**`, `scripts/build.js`, or `viewer.html` changes.

Steps:
1. Checks out repo
2. Installs dependencies (`npm ci --omit=optional`)
3. Runs `npm run build`
4. Copies `dist/index.json` → `index.json` and commits it back (`[skip ci]`)
5. Optionally deploys to remote server via rsync/SSH if secrets are configured

### Server deploy secrets (optional)

Set these in GitHub repo settings → Secrets and variables → Actions:

| Secret | Value |
|---|---|
| `NOTES_DEPLOY_KEY` | SSH private key |
| `NOTES_DEPLOY_HOST` | Server hostname |
| `NOTES_DEPLOY_USER` | SSH username |
| `NOTES_DEPLOY_PATH` | Remote path (e.g. `/home/user/`) |

If `NOTES_DEPLOY_KEY` is not set, the deploy step is silently skipped.

### validate-notes.yml — triggers on PRs touching notes/

Runs `scripts/validate-notes.js` on every PR that adds or modifies files in `notes/`.

Checks:
- File is at `notes/{author-slug}/YYYY-MM-DD.md`
- Frontmatter has `tags`, `author`, and `date`
- Date in frontmatter matches the filename
- Tags are from the approved list (max 4)
- Note body is not empty

### CODEOWNERS

`@Nili-L` is auto-assigned as reviewer on every PR via `.github/CODEOWNERS`.

## 10. Local Development

### Build the site

```bash
npm run build
```

Output goes to `dist/`. Generates `index.json`, `robots.txt`, and (if `SITE_URL` is set) `sitemap.xml`.

### Build with sitemap

```bash
SITE_URL=https://levinelabs.co.il/100devs/notes npm run build
```

### Preview in browser

Since `viewer.html` fetches `index.json` from the same directory, serve the repo root with Python:

```bash
python3 -m http.server 8080
```

Then open: `http://localhost:8080/viewer.html`

The server can also run in the background:
```bash
python3 -m http.server 8080 &
# Stop it later with:
kill %1
```

### Retag all notes

If the approved tag list changes or a tag needs to be renamed across all notes:
```bash
node scripts/retag-notes.js --dry-run  # preview
node scripts/retag-notes.js            # apply
npm run build
```

## 11. Approved Tags

Every note must have 1–4 tags from this list. All notes should include `the-hunt`.

| Tag | Use when the huddle covers... |
|---|---|
| `the-hunt` | General job search (include on every note) |
| `resume` | Resumes, cover letters, applications, ATS |
| `networking` | Coffee chats, LinkedIn, cold outreach, meetups, referrals |
| `interview` | Interview prep, behavioral questions, STAR, leetcode, system design |
| `freelance` | Freelancing, clients, proposals, pricing, contracts |
| `salary` | Salary negotiation, compensation, equity, benefits |
| `technical` | JavaScript, CSS, HTML, React, Node.js, APIs, deployment |
| `portfolio` | Portfolio projects, open source, side projects |
| `mindset` | Motivation, imposter syndrome, burnout, consistency |
| `branding` | Social media, personal brand, content creation, blogging |

To propose a new tag: mention it in your PR description. Maintainer (`@Nili-L`) will add it to the approved list and to `scripts/validate-notes.js`.
